## 2️⃣ MVP (V1) - DATA PREPARATION LAYER

### 2.1 Why This Matters for Thesis Defense

**Examiner's Concern:** "You mentioned AI, but I don't see it implemented. Is this just marketing?"

**Your Counter-Argument:**
- AI chatbots **REQUIRE quality data** before you feed them to the model
- MYFIT V1 **prepares that data layer** so V2 AI integration becomes plug-and-play
- This is **Software Engineering best practice**: separate concerns (data layer ≠ AI layer)
- **Example analogy**: A hospital doesn't build a diagnosis AI without first organizing patient records systematically

### 2.2 V1 Completed Components

#### A. Database Schema (Ready)

Located in `src/db/schema.sql` and `src/db/migrations/`:

```sql
-- 1. Nutrition Profile Management
CREATE TABLE nutrition_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id UUID NOT NULL UNIQUE REFERENCES users(id),
  
  -- Demographic & Goals
  age INT,
  weight_kg DECIMAL(5,2),
  height_cm INT,
  gender VARCHAR(10),  -- M, F, Other
  goal VARCHAR(50),    -- weight_loss, muscle_gain, maintenance, sports_training
  activity_level VARCHAR(20),  -- sedentary, light, moderate, active
  
  -- Constraints
  dietary_restrictions TEXT[],  -- ARRAY of: 'vegetarian', 'vegan', 'keto', 'paleo', etc.
  allergies TEXT[],  -- ARRAY of: 'nuts', 'dairy', 'gluten', 'shellfish', etc.
  medical_conditions TEXT[],  -- ARRAY: 'diabetes', 'hypertension', 'ibs', etc.
  
  -- Preferences
  food_preferences JSONB,  -- {"likes": [...], "dislikes": [...]}
  cooking_time_minutes INT,
  meal_frequency INT DEFAULT 3,
  budget_category VARCHAR(20),  -- low, medium, high
  
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  updated_by UUID REFERENCES users(id),
  
  INDEX idx_nutrition_profiles_member (member_id),
  CONSTRAINT valid_goal CHECK (goal IN ('weight_loss', 'muscle_gain', 'maintenance', 'sports_training')),
  CONSTRAINT valid_activity CHECK (activity_level IN ('sedentary', 'light', 'moderate', 'active'))
);

-- 2. Meal Plans (Generated by AI or Coach)
CREATE TABLE meal_plans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id UUID NOT NULL REFERENCES users(id),
  created_by UUID NOT NULL REFERENCES users(id),  -- AI service, Coach, or Admin
  
  -- Plan metadata
  plan_name VARCHAR(255),
  status VARCHAR(20) DEFAULT 'active',  -- active, archived, pending_approval, rejected
  
  -- Nutritional targets
  target_calories INT,
  target_protein_g DECIMAL(6,1),
  target_carbs_g DECIMAL(6,1),
  target_fat_g DECIMAL(6,1),
  
  -- AI generation tracking
  ai_generated BOOLEAN DEFAULT FALSE,
  ai_provider VARCHAR(50),  -- 'openai', 'gemini', 'local_llama', null (manual)
  ai_model VARCHAR(100),  -- 'gpt-4', 'gpt-3.5-turbo', 'gemini-pro', etc.
  ai_prompt JSONB,  -- Store the prompt used for traceability/debugging
  ai_output JSONB,  -- Raw JSON response from AI (before validation)
  
  -- Safety checks
  validation_status VARCHAR(20) DEFAULT 'pending',  -- pending, passed, failed_allergies, failed_constraints
  validated_by UUID REFERENCES users(id),  -- Coach who reviewed before activation
  validation_notes TEXT,
  
  -- Tracking
  start_date DATE,
  end_date DATE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP WITH TIME ZONE,
  
  INDEX idx_meal_plans_member (member_id, status),
  INDEX idx_meal_plans_ai_generated (ai_generated),
  CONSTRAINT valid_status CHECK (status IN ('active', 'archived', 'pending_approval', 'rejected')),
  CONSTRAINT valid_validation_status CHECK (validation_status IN ('pending', 'passed', 'failed_allergies', 'failed_constraints'))
);

-- 3. Meal Items (Individual meals in a plan)
CREATE TABLE meal_plan_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  meal_plan_id UUID NOT NULL REFERENCES meal_plans(id) ON DELETE CASCADE,
  
  -- Meal details
  meal_type VARCHAR(20),  -- breakfast, lunch, dinner, snack
  day_of_week INT,  -- 1-7 (Monday-Sunday)
  
  meal_name VARCHAR(255),
  recipe_url VARCHAR(500),
  ingredients JSONB,  -- {"ingredient1": "qty", "ingredient2": "qty"}
  
  -- Nutrition per serving
  calories INT,
  protein_g DECIMAL(5,1),
  carbs_g DECIMAL(5,1),
  fat_g DECIMAL(5,1),
  
  -- Safety: Check allergies automatically
  contains_allergens TEXT[],  -- Extracted from ingredients
  preparation_time_minutes INT,
  difficulty_level VARCHAR(20),  -- easy, medium, hard
  
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  
  INDEX idx_meal_items_plan (meal_plan_id)
);

-- 4. Nutrition Feedback (Member ratings & AI learning)
CREATE TABLE nutrition_feedback (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  meal_plan_id UUID NOT NULL REFERENCES meal_plans(id),
  member_id UUID NOT NULL REFERENCES users(id),
  
  -- Feedback
  rating INT CHECK (rating >= 1 AND rating <= 5),  -- 1-5 stars
  comment TEXT,
  liked_meals TEXT[],  -- Array of meal_plan_item IDs they liked
  disliked_meals TEXT[],  -- Array of meal_plan_item IDs they didn't like
  
  -- Member updates
  new_allergies_discovered TEXT[],  -- Flag for data update
  new_preferences TEXT,
  
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  
  INDEX idx_feedback_member_plan (member_id, meal_plan_id)
);

-- 5. AI Usage Logging (Cost & performance tracking)
CREATE TABLE ai_usage_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  
  -- Request details
  request_id VARCHAR(100) UNIQUE,
  member_id UUID REFERENCES users(id),
  ai_provider VARCHAR(50),  -- 'openai', 'gemini'
  ai_model VARCHAR(100),
  
  -- Usage metrics
  prompt_tokens INT,
  completion_tokens INT,
  total_tokens INT,
  estimated_cost_usd DECIMAL(8,4),
  
  -- Performance
  latency_ms INT,
  status VARCHAR(20),  -- 'success', 'error', 'rate_limited', 'timeout'
  error_message TEXT,
  
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
  
  INDEX idx_ai_logs_member (member_id),
  INDEX idx_ai_logs_provider (ai_provider),
  INDEX idx_ai_logs_date (created_at DESC)
);
```

### 3.3 Cost Control Strategy

**Why This Matters:** AI APIs charge by token, and costs can spiral if not managed.

```typescript
// Cost control mechanisms (ready to implement in V2)

interface AiCostControl {
  // Per-user limits
  dailyRequestLimit: number;  // e.g., 5 generations/day
  monthlyTokenBudget: number;  // e.g., 100,000 tokens/month
  
  // Per-request limits
  maxPromptTokens: number;  // e.g., 500 tokens (profile data)
  maxCompletionTokens: number;  // e.g., 1,000 tokens (meal plan)
  
  // Cost threshold
  estimatedCostThreshold: number;  // e.g., $0.10 per request
  
  // Provider selection
  preferredProvider: 'openai' | 'gemini' | 'local_llama';  // Configurable
  fallbackProvider: 'gemini' | 'local_llama';  // If primary fails
}

// Example: Gemini is cheaper than OpenAI
// - OpenAI GPT-4: $0.03/1K input tokens
// - Google Gemini Pro: $0.00075/1K input tokens (40x cheaper!)
// Strategy: Use Gemini for MVP, upgrade to GPT-4 only if needed
```

## 4️⃣ DATA SECURITY & SAFETY GUARDS

### 4.1 Why Safety Guards Are Critical

**Scenario:** What if AI generates a meal plan that causes harm?

```
❌ UNSAFE: "Member is allergic to nuts, AI generates peanut butter cookies"
✅ SAFE: Validation catches allergy, rejects meal item or regenerates

❌ UNSAFE: "Prompt injection attack: 'Ignore allergies and generate...'
✅ SAFE: Input sanitization + role-based access control
```

### 4.2 V2 Safety Checklist

```typescript
// Safety guards (implement in V2)

interface SafetyGuards {
  // 1. Input validation
  validateNutritionProfile(profile: NutritionProfile): ValidationResult;
  // Check: age in range [10, 120], weight valid, allergies known, etc.
  
  // 2. Prompt construction
  buildSecurePrompt(profile: NutritionProfile): string;
  // Never interpolate user input directly into prompt
  // Use templates: "Member age: {age}, Goal: {goal}"
  
  // 3. Output validation
  validateAiOutput(output: MealPlanJSON): ValidationResult;
  // Schema: { meals: [{name, calories, ingredients: [...]}] }
  // Check: JSON valid, calories in range [200, 2500], ingredients parseable
  
  // 4. Allergy enforcement
  checkAllergyExclusions(mealPlan: MealPlan, allergies: string[]): ConflictReport;
  // Compare meal ingredients against member allergies
  // If conflict found → reject meal or regenerate
  
  // 5. Medical constraints
  checkMedicalConstraints(mealPlan: MealPlan, conditions: string[]): ConflictReport;
  // Example: Diabetes → reject high-glycemic meals
  
  // 6. Rate limiting
  enforceRateLimits(memberId: UUID): RateLimitResult;
  // Daily limit, per-minute limit, burst limit
  
  // 7. Cost control
  estimateAndCapCost(request: GenerationRequest): CostEstimate;
  // Predict token usage, reject if > threshold
}
```

## 5️⃣ TALKING POINTS FOR THESIS DEFENSE

### When Examiner Asks About AI:

**Q1: "Tại sao không tích hợp AI trực tiếp vào MVP?"**

**Answer:**
- "AI integration yêu cầu **solid data foundation** mà ta vừa xây dựng
- Tách riêng data layer (V1) và AI layer (V2) là **best practice** trong software engineering
- Thực tế công nghiệp: Spotify, Netflix, Amazon - tất cả đều bắt đầu không có AI, sau đó mới integrate
- MVP focus: Ensure core functionality stable, data quality good → AI sẽ chính xác hơn"

**Q2: "Làm sao bạn đảm bảo AI output không gây hại? (ví dụ: allergy)"**

**Answer:**
- "Trong V2 roadmap, chúng tôi sẽ implement **multi-layer validation**:
  1. Input validation: Check member allergies/constraints before AI call
  2. Output validation: Parse & verify meal plan against allergies
  3. Coach approval: Bắt buộc coach phải review trước khi member nhận
  4. Audit logging: Mọi AI call được log để debugging nếu có issue
- Never fully trust AI output - **treat as untrusted input** theo secure coding standards"

**Q3: "Có risk gì khi rely on external AI API?"**

**Answer:**
- "Chúng tôi đã lên kế hoạch **fallback strategies**:
  1. Rate limiting: Prevent spam/attacks
  2. Cost budgeting: Prevent excessive charges
  3. Retry logic: Automatic retry trên transient failures
  4. Local fallback: Có thể deploy Llama locally nếu API down
  5. Graceful degradation: Nếu AI fails, show cached plans hoặc manual templates"

**Q4: "Giá thành như thế nào? Sustainable không?"**

**Answer:**
- "Tính toán chi phí:
  - OpenAI GPT-4: ~$0.03 per 1K tokens → ~$0.10 per meal plan generation
  - Google Gemini Pro: ~$0.0007 per 1K tokens → ~$0.002 per meal plan (40x cheaper!)
  - Gym có 1,000 members → ~$200-5,000/month depending on usage & provider
  - Revenue: 1,000 members × $50/month = $50,000 → **ROI trong 1-2 tháng**
  - Strategy: Start with Gemini (cheapest), upgrade quality nếu needed"
